Practical 4 Theory: Berkeley Algorithm for Clock Synchronization
Aim

To implement the Berkeley Algorithm for synchronizing the clocks of different systems in a distributed environment.

1. Introduction to Clock Synchronization

In a distributed system, many computers (called nodes) work together.
Each computer has its own internal clock.

Because of:

Clock drift
Hardware differences
Processing delays
Network delays

the clocks of different systems may show slightly different times.

Why Synchronization is Important?

Clock synchronization is required for:

Coordinating distributed processes
Ordering events correctly
Maintaining consistency in logs
Distributed databases
Transaction systems
2. What is Berkeley Algorithm?

The Berkeley Algorithm is a clock synchronization algorithm proposed by the researchers at the University of California, Berkeley.

It is used when:

There is no external UTC or NTP time server.
One machine acts as a master (time coordinator).
Other machines act as slaves (clients).

The master collects the times of all systems, computes an average, and sends the required adjustment to each client.

3. Need for Berkeley Algorithm

Suppose three computers show:

Computer	Time
Master	10:00:10
Client 1	10:00:20
Client 2	09:59:55
Client 3	10:00:05

All systems are different.

Berkeley Algorithm calculates the average time and synchronizes all systems to that average.

4. Basic Working Principle
Select one system as the master.
Master polls all clients for their current time.
Clients send their local times.
Master calculates the average time.
Master computes adjustment values.
Master sends adjustments to all clients.
Clients update their clocks.
5. Berkeley Algorithm Architecture
          Client 1
              |
              v
Client 2 --> Master <-- Client 3
              |
              v
          Client 4

The master is responsible for synchronization.

6. Step-by-Step Example

Suppose the times are:

Node	Time
Master	100 sec
Client 1	102 sec
Client 2	98 sec
Client 3	101 sec
Average Time
4
100+102+98+101
	​

=100.25 sec

4
100+102+98+101
	​

=100.25

Adjustments
Node	Adjustment
Master	+0.25 sec
Client 1	−1.75 sec
Client 2	+2.25 sec
Client 3	−0.75 sec

After applying these adjustments, all clocks become synchronized.

7. Features of Berkeley Algorithm
Centralized time coordinator (master).
Does not require external UTC.
Uses average time.
Reduces the effect of faulty clocks.
Suitable for local distributed systems.
8. Advantages
No external time source required.
Easy to implement.
Good for LAN-based distributed systems.
Handles small clock drifts effectively.
9. Disadvantages
Single point of failure (if master fails).
Accuracy depends on network delay.
Not ideal for very large networks.
Requires periodic synchronization.
10. Applications
Distributed databases
Cluster computing
Banking systems
Log synchronization
Sensor networks
11. Comparison with NTP
Feature	Berkeley Algorithm	Network Time Protocol (NTP)
External UTC required	No	Yes
Time source	Master average	Internet time servers
Suitable for	Internal networks	Internet/global systems
Complexity	Simple	More complex
12. Algorithm Steps (Exam Format)
Choose one node as the master.
Master requests time from all clients.
Clients send their current times.
Master computes the average.
Master calculates individual adjustments.
Master sends adjustments to clients.
Clients update their clocks.

The Berkeley Algorithm synchronizes clocks in a distributed system by collecting the local times of all machines, computing their average, and sending correction values to each node. It provides consistent system time without depending on an external time server.
